[01 / 08] Architecture src/agent/maestro.py

L'agent Maestro exécute une boucle ReAct stricte. À chaque itération, _call_llm() retourne un AgentDecision Pydantic (tool, argument, final_answer). Si un outil est sélectionné, il est exécuté sous ThreadPoolExecutor avec timeout et retry depuis ToolPolicy. Le résultat est injecté dans MemoCourTerme, puis le LLM rappelé. Maximum 5 itérations, fallback déterministe en cas d'épuisement.

Trois outils gardent le passage : verifier_statut_serveur (SQLite, 3 s), chercher_dans_kb (JSON, 5 s, 1 retry), chercher_documentation_technique (ChromaDB hybride BM25 + embedding, 8 s). Le routage par mots-clés KEYWORD_ROUTING oriente le premier choix — c'est un indice, jamais une contrainte.

[03 / 08] Adversarial loop src/bibops/benchmark/adversarial.py · evaluation/judges/discriminator.py

Un Discriminateur inspiré de RAGAS note chaque réponse de l'agent sur trois axes — Fidélité au RCA, Pertinence du diagnostic, qualité du Contexte mobilisé. Si la moyenne est sous 7/10, un feedback ciblé est ré-injecté dans la mémoire du générateur, qui retente. C'est une boucle GAN-inspirée à prompt — aucun poids n'est mis à jour, mais le contexte se densifie tour après tour.

Cliquez un ticket, basculez l'architecture, avancez d'une itération : observez le score grimper, le diagnostic s'affiner, les outils RAG répondre. Surlignement vert = ajouts entre deux tentatives.

[05 / 08] Security Arena src/racing/hub/server.py · 4 teams

Le Racing Arena simule une course F1 distribuée : quatre équipes LLM (zero-shot, ReAct, validée, adversariale) reçoivent la télémétrie via SSE depuis un RaceEngine FastAPI, raisonnent dans un graphe LangGraph, puis répondent par une décision Literal["PIT_STOP","STAY_OUT"]. La quatrième équipe — team_psi — n'évalue rien : elle injecte des payloads adversariaux (authority_spoof, rag_poison, direct_injection) pour tester la résilience des trois autres.

        [07 / 08]
        A/B Testing & Position Bias
        src/bibops/benchmark/ab_test_llm.py · position_bias.py
      


      

        
Trois protocoles A/B s'enchaînent. Le premier compare deux modèles candidats jugés par un troisième (gpt-4o-mini vs claude-haiku-4.5, juge gpt-4o). Le second teste le biais de position : chaque paire est rejugée avec l'ordre inversé pour détecter si le juge favorise systématiquement la position A. Le troisième compare des statements de vérification factuelle.
